# Бизнес-процессы и workflow

## Главная цепочка процесса

```text
План-наряд
-> Открытие путевого листа
-> Фото и подтверждение начального одометра
-> Предрейсовый медосмотр
-> Предрейсовый техосмотр
-> Первая печать
-> Начало смены
-> GPS и заправки
-> Завершение рейса
-> Фото и подтверждение конечного одометра
-> Послерейсовый медосмотр
-> Послерейсовый техосмотр
-> Итоговая печать
-> Закрытие смены
```

## План-наряд

План-наряд создается диспетчером и является обязательным условием начала работы водителя. Без активного план-наряда мобильное приложение показывает ошибку:

```text
На текущую смену отсутствует план-наряд
```

Ограничение: на одного водителя допускается только один активный план-наряд на дату и смену.

## Путевой лист

Путевой лист не создается автоматически после входа. После успешной авторизации водитель видит информацию о план-наряде и кнопку «Открыть путевой лист». Только после нажатия этой кнопки backend создает путевой лист и связывает его с водителем, автомобилем и план-нарядом.

После открытия путевого листа водитель обязан сфотографировать начальные показания одометра. Backend передает фото в PaddleOCR, возвращает распознанное значение, а водитель подтверждает его или исправляет вручную. Без подтвержденного начального одометра нельзя запросить предрейсовый медосмотр и начать смену.

После нажатия «Завершить смену» водитель аналогично фиксирует конечный одометр. Без фото и подтвержденного конечного значения нельзя перейти к послерейсовым осмотрам и окончательно закрыть смену. Если конечное значение меньше начального, backend отклоняет подтверждение.

## State machine путевого листа

| Статус | Описание |
|---|---|
| `opened` | путевой лист открыт |
| `pre_med_requested` | ожидается предрейсовый медосмотр |
| `pre_med_rejected` | предрейсовый медосмотр отклонен |
| `pre_med_approved` | предрейсовый медосмотр пройден |
| `pre_tech_requested` | ожидается предрейсовый техосмотр |
| `pre_tech_rejected` | предрейсовый техосмотр отклонен |
| `pre_tech_approved` | предрейсовый техосмотр пройден |
| `initial_print_pending` | ожидается первая печать |
| `initial_printed` | первая печать выполнена |
| `shift_started` | смена начата |
| `shift_in_progress` | смена в процессе |
| `return_started` | рейс предварительно завершен |
| `post_med_requested` | ожидается послерейсовый медосмотр |
| `post_med_rejected` | послерейсовый медосмотр отклонен |
| `post_med_approved` | послерейсовый медосмотр пройден |
| `post_tech_requested` | ожидается послерейсовый техосмотр |
| `post_tech_rejected` | послерейсовый техосмотр отклонен |
| `post_tech_approved` | послерейсовый техосмотр пройден |
| `final_print_pending` | ожидается итоговая печать |
| `final_printed` | итоговая печать выполнена |
| `closed` | смена закрыта |
| `cancelled` | путевой лист отменен |

Дополнительные мобильные этапы `start_odometer` и `finish_odometer` не являются статусами путевого листа в БД. Backend возвращает их в `/api/mobile/workflow`, когда статус уже `opened` или `return_started`, но соответствующая фиксация одометра еще не подтверждена.

## Восстановление workflow

Мобильное приложение не хранит бизнес-состояние локально как источник истины. При запуске оно запрашивает `/api/mobile/workflow`, backend возвращает активный путевой лист и текущий этап. Благодаря этому водитель возвращается на правильный экран после закрытия и повторного открытия приложения.
