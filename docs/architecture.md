# Архитектура системы

## Общая схема

Информационная система ООО «АЗЫК» строится как распределенная клиент-серверная система. Пользовательские интерфейсы вынесены в два отдельных клиента:

- мобильное приложение водителя;
- web-административная панель сотрудников.

Оба клиента взаимодействуют с единым backend API. Backend содержит всю бизнес-логику, правила доступа, обработку состояний путевого листа, генерацию PDF, отчеты и WebSocket-события.

## Компоненты

| Компонент | Технология | Назначение |
|---|---|---|
| Backend API | Laravel | единая точка бизнес-логики и данных |
| Mobile app | Flutter | пошаговый workflow водителя |
| Admin panel | Vue 3 + Pinia | административная работа сотрудников |
| Database | PostgreSQL | хранение учетных данных и предметных сущностей |
| Cache/Queue/WS | Redis | очереди, кэш, события реального времени |
| PDF | Blade + Dompdf | формирование путевых листов |
| Maps | OpenStreetMap + Leaflet | карта текущих координат и история движения |
| OCR | Python + PaddleOCR | распознавание показаний одометра по фотографии |

## Почему монолитный backend

Для дипломного проекта и внутренней системы предприятия монолитный Laravel API является наиболее рациональным решением:

- проще разработка, тестирование и развертывание;
- вся бизнес-логика находится в одном месте;
- нет необходимости в сложной инфраструктуре микросервисов;
- роли, осмотры, путевые листы и отчеты тесно связаны между собой;
- объем системы соответствует одному backend-приложению.

## Внутренние backend-модули

| Модуль | Ответственность |
|---|---|
| Auth | вход, выход, токены, проверка канала доступа |
| Users | пользователи, роли, смена паролей |
| Drivers | карточки водителей |
| Vehicles | карточки автомобилей |
| WorkOrders | план-наряды |
| Waybills | путевые листы и state machine |
| Inspections | медосмотры и техосмотры |
| Fuel | заправки |
| GPS | координаты, текущая карта, история движения |
| Reports | отчеты и Excel |
| PDF | генерация путевых листов |
| Audit | журнал действий |
| OdometerControl | фото одометра, OCR-результат, подтвержденные значения и сравнение с GPS |

OCR-компонент вынесен в отдельный Docker-контейнер только как модуль инференса нейросетевой модели. Он не содержит бизнес-логики, не меняет состояния путевого листа и не записывает пробег в БД. Laravel остается главным backend-приложением: принимает фото, вызывает OCR, сохраняет результат и ждет подтверждения водителя.

## Авторизация

Используется Laravel Sanctum. Выбор Sanctum объясняется тем, что система имеет собственные клиенты и не требует внешнего OAuth. Sanctum проще Laravel Passport, поддерживает API-токены для мобильного приложения и SPA-админки, хорошо интегрируется с Laravel middleware и policies.

## Разделение доступа по клиентам

| Канал | Разрешенные роли |
|---|---|
| Mobile app | только `driver` |
| Admin panel | `admin`, `dispatcher`, `medic`, `mechanic` |

Водитель не должен авторизоваться в web-админке. Сотрудник не должен авторизоваться в мобильном приложении.
